CST / UTC / GMT 的区别

Unix时间戳与当前时区无关

  1. 定义为从格林威治时间1970年01月01日00时00分00秒到现在的秒数。因为同一时刻，不同时区得到的时间戳都是一致，比如1452783600这个时间戳，在巴黎是下午4点，在旧金山就是早上7点。（可以这样理解：之所以同一个时间戳在不同时区转化的时间不同，是因为当时间戳为0时，各个时区当时的时间都是不同的啊，而格林威治刚好是1970/01/01/00:00:00）
  2. 因此采用时间戳的方式存储时间，就可以跨时区显示当地的时间
  3. python中，只要使用time.localtime(时间戳)就可以获得这个时间戳在当地的显示时间。例如time.localtime(0)显示就是1970/01/01/08:00:00
  4. 但时间戳的缺点是只有4个字节，2038年就不能用了

Mysql 时间问题扫盲

  1. timestamp 和 datetime 都是时间相关的类型。采用字符串的形式插入（但是数据库会对这个字符串的合法性进行检查，正确时才能插入）：”2017-11-11 16:16:24”。 或者采用sql的now()方法获得当前数据库时区的时间字符串。
  2. datetime存入的时间是时区无关的，读取的时间字符串就是存入的时间字符串。(比如插入“2000-12-12 10:23:24”后，调整数据库的时区，再读取数据还是这个字符串) [猜测，内部还是转化成时间戳了，但是其默认时区是格林威治，因此无论实际数据库的时区怎样设置都不会有影响。]
  3. timestamp 如其名称所示，内部其实存储的时间戳，因此可以做到时区相关显示。（比如插入“2000-12-12 10：23:34”时，其实mysql服务器内部按照当前的时区转成了时间戳再存入，因此切换数据库时区后再读取该值时，数据库会先将该时间戳转化为当前时区的时间再显示）
  4. timestamp 只占4个字节，而 datetime 占8字节
  5. timestamp 默认非NULL，用户不填，而默认调用now()获得当前时区的时间插入。 而datetime可以是NULL
  6. 结合上述：只要当读和写数据库时的数据库服务器的时区没有改变，timestamp 和 datetime 就没有区别。但是其实很容易发生改变（这里并不是指服务器全局的时区设置改变，一般不会去改），而是当我们应用程序连接数据库时，可以设置这条连接会话的时区。比如我在北美的应用服务器，连接中国的数据库时，可以在连接上设置北美的时区，这样北美的应用服务器向中国的数据库服务器发起读写操作时的时区都是指定为北美的时区（一般也是用这种方法，实现时间的跨区域显示!）。因此timestamp 比 datetime 好用！
  7. ORM 作为数据库原始数据的消费者，无论如何设置ORM的时区参数，都无法改变datetime是时区无关的缺点
