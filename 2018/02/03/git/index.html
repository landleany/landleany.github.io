GIT 分支

  1. git branch -a 查看所有分支
  2. git branch -r 查看远程分支
  3. git branch 查看本地分支
  4. git branch test 创建分支
  5. git branch -d test 删除本地分支
  6. git checkout test 切换分支
  7. git checkout -b test 创建并切换到该分支
  8. git remote add name git-url 添加远程仓库并设置别名
  9. git remote 可以查看当前有哪些远程仓库
  10. git fetch 远程仓库名 获取该远程仓库所有分支并在本地建立对应的本地分支
  11. git push 远程仓库别名 分支名 推送分支，可以用来创建对应的远程分支
  12. git push –delete branch_name 删除远程分支
  13. git push –delete tag tag_name 删除远程 tag
  14. git pull –tag 才能把远程的 tag 拉到本地
  15. git clean -f 删除 untrack files
  16. git clean -fd 删除 untrack files and dirs
  17. git reset –hard ORIG_HEAD 回退到 pull 之前的 commit，会丢失所有修改
  18. git config –global user.name “John Doe” 设置全局用户名 去掉 global 表示针对项目设置
  19. git config –global user.email johndoe@example.com 设置全局email 去掉 global 表示针对项目设置

GIT fetch 用法

一旦远程主机的版本库有了更新（Git术语叫做commit），需要将这些更新取回本地，这时就要用到git fetch命令。git fetch命令通常用来查看其他人的进程，因为它取回的代码对你本地的开发代码没有影响。默认情况下，git fetch取回所有分支（branch）的更新。如果只想取回特定分支的更新，可以指定分支名。然后也可以使用git merge命令或者git rebase命令，在本地分支上合并远程分支。

GIT pull 用法

git pull命令的作用是，取回远程主机某个分支的更新，再与本地的指定分支合并（等同于先fetch，再merge）。默认使用 git merge 来合并，也可以用 –rebase 指定使用 git rebase 合并分支。

如果远程分支是与当前分支合并，则冒号后面的部分:<本地分支名>可以省略。
如果直接输入 git pull，则会先 git fetch，即拉取所有远程分支的更新进本地的远程分支，然后单独 merge 当前分支与其本地的远程分支。
只有当本地分支比远程分支超前时，才能成功 push

pull request (pr) 和 merge request (mr) 区别

对于项目的核心成员，集中式版本管理和分布式版本管理贡献代码的方式并没有多大差异（这里不要纠结个人使用层面的差异，只谈论为仓库贡献代码的方式）。但对于非项目核心成员来说，集中式的版本管理就非常痛苦了，因为他们找不到方式来提交自己的代码（请忽略低效的发邮件补丁吧……）。然而分布式版本管理则解决了这个问题：非项目核心成员可以克隆仓库，这样就得到了一个自己具有完全读写权限的仓库，贡献的代码可以完全同步到这个具有完全读写权限的仓库中。

为了让非核心成员提交的代码被核心成员接纳，非核心成员会向核心成员提出“申请（Request）”去自己的仓库指定分支中“拉取(pull)”最新的修改，这便是 Pull Request 的来源。

那么 Merge Request 又是什么呢？GitLab 对此的解释是—— 一样的 ，没有区别。Merge 只是在强调最后的那个动作“合并（Merge）”。

GitHub、Bitbucket 和码云（Gitee.com）选择 Pull Request 作为这项功能的名称
GitLab 和 Gitorious 选择 Merge Request 作为这项功能的名称

GIT 取消已经暂存（git add）的文件

git reset HEAD somefile
该文件进入到未暂存阶段，但修改还在，如果还要恢复修改
git checkout – somefile

GIT 版本回退并同步全程仓库

大概有两种方法：reset 和 revert。首先说下结论，尽量用 revert。

  1. reset，直接移动 HEAD 指针，例如 git reset HEAD~2 即将 HEAD 指针移动到倒数第二次提交位置，默认为–soft参数，表示将回退过程中的修改放入工作区，而–hard则表示丢弃这些修改。然后调用 git push --forse强制更新远程分支。 这种方法不好的地方是，修改后，远程分支实际是后退了，其他人（如果之前已经同步了你回退前的状态）merge 的时候还是会将回退的部分给同步回来！
  2. revert，并不是直接移动 HEAD 指针，而是将需要回退的修改 提交为一个新的commit，这个就避免了协作开发时，其他人把回退的内容给merge回来的问题。

GIT 什么是 origin master

假设你们团队有个地址为 git.ourcompany.com 的 Git 服务器。如果你从这里克隆，Git 会自动为你将此远程仓库命名为 origin，并下载其中所有的数据，建立一个指向它的 master 分支的指针，在本地命名为 origin/master，但你无法在本地更改其数据。接着，Git 建立一个属于你自己的本地 master 分支，始于 origin 上 master 分支相同的位置。

GIT 查看远程仓库

很抱歉，没有什么直接的命令可以查看某个本地分支对应的远程分支（upstream），但可以用下面的方法查看：

  1. 直接通过 .git/config 文件查看
  2. git branch -vv 不仅可以查看对应的upstream，还可以看到最新的log，非常棒

切换其他远程分支

  1. 例如，coding.net 上建立一个仓库
  2. git remote add coding addr
  3. git push -u coding master
